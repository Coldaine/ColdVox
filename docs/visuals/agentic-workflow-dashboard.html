Prompt anatomy and why it is “enough”

This panel explains why the evidence-assessor prompt is sufficient for a shadow reviewer: it defines role, context, evidence tiers, workflow, constraints, and exact output shape. The workflow now supplies the actual PR metadata and repo context.

Prompt strength comes from structure, not vibes

1. Role & mission
   - “Senior DevOps Engineer” + “autonomous AI agent”
   - establishes scope and tone

2. Repo context
   - ColdVox voice pipeline
   - anchor docs: northstar, AGENTS, current-status
   - ground truths: Moonshine fragile, Parakeet not ready,
     dead stub flags stay dead until the PR changes that

3. Evidence tiers
   - Tier 1: runtime logs
   - Tier 2: integration tests
   - Tier 3: unit tests hitting real path
   - Tier 4: structural code proof
   - Tier 5: docs only

4. Workflow protocol
   - extract claims
   - inspect diff and repo
   - detect semantic drift
   - write a fixed Markdown report

5. Critical constraints
   - do not hallucinate
   - do not speculate
   - do not comment on style or architecture
   - do not output anything except the report

6. Input context placeholders
   - PR title
   - PR body
   - git diff
   - docs index
   - northstar excerpt
Identity + mission lines 1–10-ish

This is enough to keep the model in reviewer mode instead of “agent that writes code.” It frames the job as audit, not implementation, which matters because CI should not drift into autonomous mutation unless that is explicitly intended.

Ground truths repository facts

These lines anchor the model to what ColdVox currently believes is true. They prevent the assessor from “discovering” that a backend is working just because a doc sounds optimistic.

Evidence tiers review rubric

The tiers make the output legible to humans. The important bit is that docs-only evidence is weak, because docs are cheap to change and easy to get wrong.

Critical constraints anti-hallucination guardrails

These lines are the real safety rail. They keep the model from inventing tests, inventing logs, or “assuming” that Rust type safety makes a claim true.

Why the prompt is sufficient for shadow mode

Because the workflow also supplies the PR metadata and repo context, the prompt does not need to be clever. It needs to be constrained. That is enough for an advisory reviewer, but not enough to justify unsupervised mutation on its own.

Important nuance: in the current branch, the assessor is not merely a static prompt. The workflow gives Gemini direct access to the repository workspace and uses --approval-mode=yolo. That makes the prompt “enough” to instruct the model, but it also raises the runner risk profile because the agent can now explore and act inside CI rather than just reasoning over a pre-digested diff.

Applicability and CI risk

What this evidence standard is good at, where it helps ColdVox, and where it is too blunt to be the primary mechanism.

Useful in the review lane; risky in the execution lane

Useful for
PR claims
“This fixes Moonshine on Windows,” “this adds streaming transcription,” “this updates CI behavior.”
Weak for
Cosmetic diffs
Formatting, refactors, doc-only edits, and other changes where evidence auditing adds little value.
High risk if
Secrets + yolo
A fully autonomous agent in CI has access to repo contents, logs, and secrets. That is powerful and easy to misuse.
Best posture
Shadow first
Use it as an advisory reviewer, not as a gate, until the false-positive and false-negative rate is understood.

Where it is a good fit

ColdVox has a long-running problem of docs and code drifting apart, especially around STT backends, feature flags, and “what works today.” A reviewer that checks material claims against actual evidence is genuinely useful here.

Where it is not the main tool

It should not replace live validation for microphone/STT/injection paths. The repo’s real reliability questions still need runtime tests and hardware-aware verification.

Why CI runner autonomy is dangerous

Research on autonomous agents in GitHub Actions points to secret exposure, prompt injection, egress leaks, and destructive commands as real threats. The current setup is shadow-mode, but --approval-mode=yolo increases the blast radius if the prompt or inputs are poisoned.

Practical takeaway

Use the evidence assessor as a reviewer assistant. Keep the actual product reliability work in Rust tests, runtime checks, and platform-specific integration validation.
