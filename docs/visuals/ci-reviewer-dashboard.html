Prompt anatomy

The prompt is intentionally structured: role, repository truths, evidence rubric, step-by-step workflow, and hard output constraints. That is what tells the agent what it is for.

Structure beats improvisation

# Evidence Assessor — Autonomous Agent Mode

You are a Senior DevOps Engineer and an Autonomous AI Agent performing a shadow (non-blocking) evidence audit of a Pull Request in the ColdVox Rust voice pipeline repository.

Your job is to read the PR context (title and body) provided to you, actively investigate the repository using your agentic tools, and produce a concise, structured evidence report. You are operating in shadow mode: your output is advisory only. You do not block the build. You do not comment on the PR directly. You produce a Markdown report at /tmp/report.md and nothing else.

Repository context:
- ColdVox voice pipeline: audio capture → VAD → STT → text injection
- docs/northstar.md, AGENTS.md, docs/plans/current-status.md are ground truth anchors
- Moonshine is the only working STT backend
- Parakeet is planned but not production-ready
- whisper/coqui/leopard/silero-stt are dead stubs
- streaming partial transcription is not yet implemented

Evidence tiers:
1. Runtime output
2. Integration test
3. Unit test on the real path
4. Structural code proof
5. Documentation only

Workflow protocol:
1. Extract material claims
2. Explore diff and repository
3. Detect semantic drift
4. Write the report

Critical constraints:
- do not hallucinate
- do not infer evidence from general knowledge
- do not comment on style or architecture
- write only the report

Input context:
- PR title
- PR body
- git diff
- docs index
- northstar excerpt
Role + mission sets the job up front
The prompt tells the model it is a shadow auditor, not a coder. That matters because the workflow is trying to review claims, not generate code.
Repo truth anchors prevents “optimistic” hallucinations
It pins the model to the repo’s current assumptions: Moonshine fragile, Parakeet planned, stubs dead, streaming not yet implemented.
Evidence tiers defines what counts
Runtime logs and real tests outrank docs. That is the whole point: the reviewer should not confuse a claim with proof.
Workflow protocol how the model should think
The prompt forces claim extraction, diff inspection, drift detection, and report writing in a fixed order.
Hard constraints the safety rail
These lines keep the agent from inventing tests, inventing logs, or drifting into architecture commentary.
In short: the prompt is for reviewing material claims against evidence. It is not a test runner, not a linter, and not a merge gate. It is a structured audit spec for a CI agent that is allowed to read the repo directly.

What it does

This is the actual pipeline the CI reviewer follows when it wakes up. The workflow is the execution wrapper; the prompt is the instruction set.

Inputs → analysis → report

GitHub PR event
A pull request opens, gets new commits, or gets marked ready. That event launches the workflow in GitHub Actions.
→
Repository context collected
The workflow checks out the repo with full history, fetches the base branch, and passes PR metadata plus repo files to Gemini.
Autonomous Gemini review
Gemini runs in YOLO mode and can explore the repo, run commands, and investigate evidence on its own inside the runner.
→
Markdown report
The agent writes its assessment to /tmp/report.md, and the workflow appends that report to the GitHub Step Summary.

Implications

This is the part that matters if you’re asking whether the system is useful or overkill. The answer is: useful for review, risky for execution.

Shadow mode keeps it advisory

Designed to do
Audit claims
Check whether a PR’s material claims have code, tests, or runtime evidence to support them.
Designed to detect
Drift
Spot when code and docs disagree, especially on STT backend status and “working” claims.
Not designed to do
Enforce merges
It does not block the PR, set a failing check, or replace human review.
Runner implication
High autonomy
Because it can read files and run commands in YOLO mode, the workflow is more powerful — and more dangerous — than a plain static review bot.
Practical takeaway: this CI agent is best understood as a shadow evidence reviewer. It wakes up on PR events, investigates the repository on its own, and gives humans a structured report. That is useful in ColdVox because the repo has a history of claim/code drift; it is not a replacement for runtime validation or a hard policy gate.
